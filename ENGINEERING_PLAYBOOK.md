# دليل البناء الهندسي المعتمد — Engineering Playbook

> طريقة عملنا القياسية لبناء مواقع ومنصّات الويب. مبنية على مشروع **عبور** كتطبيق مرجعي.
> الهدف: نفس الجودة، وقت أقل، أخطاء لا تتكرّر. هذا الملف هو المرجع الملزِم للفريق.

---

## 1. الستاك المعتمد (Canonical Stack)

| الطبقة | الاختيار | ملاحظة |
|--------|----------|--------|
| الواجهة | **Next.js 16** (App Router, React 19) + TypeScript | SSR/ISR، توجيه ملفّي |
| التنسيق | **Tailwind CSS v4** | خصائص منطقية لدعم RTL |
| الباك إند | **Django 5 + DRF** | admin جاهز، ORM ناضج |
| قاعدة البيانات | **PostgreSQL 17** | ذاتية الاستضافة |
| التخزين | **MinIO** (S3-compatible) | الكود لا يتغيّر عن S3 |
| الحاويات | **Docker + Compose** | خرج standalone للواجهة |
| الحافة | **nginx + Cloudflare** | reverse proxy + SSL/CDN |
| التعدّد اللغوي | نظام i18n مخصّص | cookie locale + `pick()` |

> ثبّت نسخ الأدوات (Node، Python، Postgres) مطابِقة للإنتاج. اختلاف النسخة = مفاجأة عند النشر.

## 2. خط سير البناء — 7 مراحل بالترتيب

الترتيب مقصود. **لا تنتقل لمرحلة قبل استيفاء معايير قبول السابقة.**

### المرحلة 0 — ترجمة الديزاين (Figma → Code)
- [ ] استخراج **رموز التصميم** (ألوان، مسافات، خطوط، نصف أقطار) كمصدر واحد → `tailwind` config.
- [ ] مكوّن لكل قسم في التصميم، بأسماء مطابقة لأسماء الفريمات.
- [ ] لا ألوان/مقاسات «سحرية» في الكود — كلها من الرموز.

### المرحلة 1 — الواجهة (Frontend)
- [ ] App Router + TypeScript.
- [ ] **RTL/LTR وثنائية اللغة من البداية**: كوكي اللغة، `pick(ar, en)`، اصطلاح بيانات `_EN`.
- [ ] استخدم الخصائص المنطقية (`ms-`, `me-`, `ps-`, `start`, `end`) لا (`ml`, `left`…).
- [ ] كل نص مرئي له نسختان (عربي/إنجليزي) — لا نص مكتوب مباشرة (hard-coded) بلغة واحدة.
- [ ] توحيد التسمية عبر الهيدر/الفوتر/مسار التنقّل/`H1`/الـ metadata.

### المرحلة 2 — الباك إند ونموذج المحتوى (Backend)
- [ ] Django + DRF، فصل: **الطلبات** (الفورمات) عن **المحتوى** (CMS).
- [ ] جسر `fetchContent` مع **سقوط آمن** إلى بيانات ثابتة عند تعطّل/بطء الباك إند.
- [ ] كل حقل محتوى ثنائي اللغة (`*_ar` / `*_en`).
- [ ] مهلة (timeout) على نداءات الباك إند من الواجهة — لا تعليق الرندر.

### المرحلة 3 — لوحة التحكّم (CMS)
- [ ] محرّر مبني على **schema** (عرّف الحقول → المحرّر يتولّد).
- [ ] مصادقة **بالتوكن** (لا كوكيز) — تعمل خلف أي بروكسي.
- [ ] **مبدأ المرآة**: اللوحة تعكس كل صفحة بالكامل، بترتيب الصفحة، بلا زحام ولا حقول مخفية.
- [ ] «استعادة الافتراضي» عبر لقطات (snapshots)، ونسبة اكتمال لكل قسم.
- [ ] نفس هوية الموقع (ألوان/شعار/خط) داخل اللوحة.
- [ ] responsive على الموبايل — تُختبر فعليًا.

### المرحلة 4 — دليل المستخدم (Admin Guide)
- [ ] دليل بالصور خطوة بخطوة، ثنائي اللغة، قابل للبحث والطباعة PDF.
- [ ] يغطّي كل شاشة وكل نوع عنصر (عنوان، نص، أيقونة، صورة، كرت، رقم…).
- [ ] **محمي خلف مصادقة اللوحة** — لا يراه إلا الأدمن.

### المرحلة 5 — النشر الذاتي (Deployment)
- [ ] `Dockerfile` لكل خدمة + `docker-compose.yml` + `.dockerignore`.
- [ ] `.env.example` كامل لكل خدمة (قيم وهمية)، والأسرار الحقيقية خارج git.
- [ ] بورتات الحاويات مربوطة على `127.0.0.1` فقط.
- [ ] nginx بروكسي عكسي؛ العامة يصل 80/443 فقط.
- [ ] Cloudflare للـ DNS/SSL؛ الواجهة تُبنى بمسار API نسبي `/api`.
- [ ] نسخ احتياطي مجدوَل (قاعدة البيانات + التخزين).

### المرحلة 6 — التكامل مع الـ ERP (القادم)
- [ ] حدود API واضحة لكل موديول (عقد مكتوب أولًا).
- [ ] طبقة **محوّل (adapter)** لكل موديول تعزل تفاصيل الـERP.
- [ ] مزامنة عبر أحداث/مهام مع سقوط آمن.
- [ ] ابدأ بموديول واحد → أثبت النمط → عمّم.

## 3. المبادئ الحاكمة (10)

1. **ثنائية اللغة و RTL** من اليوم الأول.
2. **اللوحة تعكس الموقع 1:1** وبنفس الهوية.
3. **رموز التصميم** مصدر واحد للحقيقة.
4. **الحاويات و`.env.example`** من الأسبوع الأول.
5. **لا سرّ في git** أبدًا — أمثلة فقط.
6. **كل واجهة أدمن خلف مصادقة** — بلا استثناء.
7. **التوثيق والدليل جزء من التسليم**.
8. **مسارات API نسبية** للقابلية للنقل.
9. **سقوط آمن** عند تعطّل الباك إند.
10. **اختبر قبل أن تقفل أو تسلّم**.

## 4. كتالوج الأخطاء (Problem → Fix)

| الخطأ | الإصلاح المعتمد |
|-------|------------------|
| تعارُض الاعتماديات في بناء Docker | أبقِ الـ lock file متزامنًا، ثبّت نسخة Node، واختبر بناء Docker **مبكّرًا**. احذر الاعتماديات الخاصة بالمنصّة (استخدم `npm install` لا `ci` عند اللزوم). |
| أسرار في المستودع | `.env.example` بقيم وهمية فقط؛ الأسرار الحقيقية تُرسَل خاصةً وتوضع على السيرفر يدويًا. |
| إعداد بيئة خاطئ (بادئة `/api`) | ملف بيئة واحد لكل خدمة، وطابِق صيغة المسارات بين الكود والبروكسي. |
| رابط قاعدة بيانات فيه محارف خاصة (`%`) | مرّر بيانات الاتصال منفصلة (host/port/user/pass) لا كـ URI. |
| اختلاف نسخة Postgres عند `pg_dump` | ثبّت نسخة أداة الـ dump = نسخة خادم الإنتاج. |
| SSL إجباري على Postgres محلي | اجعل SSL اختياريًا عبر متغيّر بيئة (`DATABASE_SSL=disable`). |
| محتوى مختلط على HTTPS | روابط أصول مطلقة بـ HTTPS على نفس الدومين، أو مسارات نسبية same-origin. |
| بورتات مكشوفة (Docker + ufw) | اربط على `127.0.0.1`؛ Docker يتجاوز ufw إن نشرت على `0.0.0.0`. |
| صفحة أدمن «مخفية» بلا حماية | بوّب كل واجهة أدمن خلف مصادقة — لا تعتمد على «الرابط سرّي». |
| تأجيل الاستضافة للنهاية | جهّز الحاويات وشغّل بناء الإنتاج دوريًا من البداية. |

## 5. مرجع النشر السريع (Deployment Runbook)

```bash
# 1) الكود على السيرفر (git clone أو نقل الأرشيف)
# 2) ملفات البيئة في أماكنها:
#    ./.env            → متغيّرات docker compose (كلمات مرور Postgres/MinIO، NEXT_PUBLIC)
#    ./.env.local      → خدمة الواجهة (frontend)
#    ./backend/.env    → خدمة الباك إند (backend)
# 3) التشغيل:
docker compose up -d --build
```

**قاعدة ملفات البيئة:** خدمة واحدة = ملف بيئة واحد. الـ compose يربط كلًّا بخدمته:

```yaml
backend:   env_file: ./backend/.env
frontend:  env_file: .env.local
# postgres / minio: تقرأ من ./.env عبر ${VAR}
```

**البنية:** `Cloudflare (SSL) → nginx:80 → [frontend:3000 | backend:8000] → Postgres + MinIO` — كلها على `127.0.0.1`.

## 6. اصطلاحات المستودع

- `.gitignore`: تجاهل `.env*` مع استثناء `!*.env.example`.
- تسمية الفروع والـ commits واضحة؛ لا أسرار في الرسائل أو التاريخ.
- كل خدمة لها `Dockerfile` و`.dockerignore` خاص بها.
- التوثيق (هذا الملف + دليل المستخدم) يُحدَّث مع كل تغيير جوهري.
